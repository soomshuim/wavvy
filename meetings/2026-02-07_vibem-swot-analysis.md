## Phase 1: 전문가별 SWOT 분석

### Product Leader

**S (강점):**
- 극도로 정교한 품질 관리 시스템 (20개 슬롯 검증, 3-Point QC, Pure Input Principle)
- 차별화된 24시간 세계관 IP — 청자가 "지금 이 순간"에 맞는 음악을 찾는 UX 동선
- 1인 운영 최적화 CLI 도구 — validate → preview → pack → shorts 워크플로우
- SSOT 문서 시스템 (3,268줄) — AI 협업 시 재현성과 품질 일관성 보장

**W (약점):**
- Suno 플랫폼 정책 변경/서비스 중단 시 전체 파이프라인 마비 가능
- 규칙 복잡도의 양날의 검 — 학습 곡선 가파름, 신규 크리에이터 진입장벽
- 유튜브 알고리즘 단일 의존 — 플랫폼 리스크 분산 전략 부재
- 1인 운영의 확장성 한계 — 팀 확장 없이는 24시간 10 Stations 완성 불가능

**O (기회):**
- AI 음악 네이티브 세대 부상 — 선점 우위 확보 가능
- 감성 플레이리스트 시장 성장 — 검증된 포맷에 차별화 레이어 추가
- 숏츠 바이럴 레버리지 — TikTok/Reels 크로스 플랫폼 확산
- B2B 라이센싱 — 카페/공간/브랜드 BGM 상품화

**T (위협):**
- AI 생성 음악 저작권 불확실성
- 감성 플레이리스트 시장 포화 — 후발주자 등장 시 차별화 약화
- YPP 반복 콘텐츠 정책 강화 위험
- 한국어 가사 시장 규모 한계 (TAM 제약)

**핵심 인사이트:** VIBE-M은 "엔지니어링된 창작" 사례. 단기 숏츠 바이럴 + 시간대 IP 선점, 중장기 라이센싱/멀티플랫폼/커뮤니티로 리스크 분산 필요.

**W (약점):**

- 리텐션 메커니즘 부재 — 구독자와의 지속적 접점 부족
- 데이터 기반 최적화 미흡 — A/B 테스트, 퍼널 추적 인프라 부재
- 바이럴 루프 설계 부족 — 자연 확산 메커니즘 미비

**O (기회):**
- 감성 AI 음악 시장 폭발 — 선점자 우위
- 시간대별 루틴 습관화 — DAU 성장 및 강력한 리텐션

- 크로스 플랫폼 확장 — Spotify, Apple Music 유입 다각화

**T (위협):**
- YouTube 알고리즘 의존성
- AI 음악 저작권 불확실성
- 경쟁자 진입 장벽 낮음
- 플레이리스트 피로도

**핵심 인사이트:** (1) Shorts→롱폼 전환 퍼널 최적화, (2) 시간대별 알림/습관화 리텐션 루프, (3) 커뮤니티 참여 기반 바이럴 메커니즘 구축 필수.

### Data Scientist

**S (강점):**
- 완벽한 트레이서빌리티 — report.json 트랙별 BPM/mood/genre/duration/SHA-256 기록
- 체계적 변수 통제 프레임워크 — 5-Slot 시스템 + Vocal Persona
- 20-Slot QC = Feature Engineering 프레임워크 — 40개+ Feature 구조화
- Fail-Fast 피드백 루프 — DEBUG 모드 1개 변수 변경 A/B 테스트

**W (약점):**
- 정량적 성과 지표 부재 — 청취 데이터(재생시간, 리텐션, CTR) 없음
- 1인 운영의 샘플링 바이어스 — 단일 주체 주관 판단 의존
- 메타데이터 구조화 미흡 — Playlist 레벨과 트랙 레벨 미연결
- 실험 재현성 한계 — Suno AI 블랙박스

**O (기회):**
- YouTube Analytics 통합 — 24H Universe 가설 실증 가능
- Causal Inference 실험 설계 — "BPM이 70-76일 때 새벽 4시 재생시간 증가?" 검증
- Shorts 데이터로 Hook Effectiveness 측정 — A/B 테스트 가능
- 한국어 감성 NLP 데이터셋 구축 — 학술적/산업적 가치

**T (위협):**
- Suno AI 의존성 — 모델 업데이트 시 성공 패턴 무효화
- YouTube 알고리즘 변동성 — 과거 데이터 유효성 상실
- 스케일업 시 재현성 붕괴 — 주관적 지표 다른 작업자 재현 불가
- 시간대 가설 검증 지연 — 실제 사용자 행동 데이터 없이 컨셉만 앞서감

**핵심 인사이트:** 가장 큰 기회는 **지금 바로 측정을 시작하는 것**. YouTube Analytics API 연결 → 재생 시간대/이탈 구간/트랙별 retention을 report.json과 통합하면 24H Universe 가설 검증 + 학술 논문급 데이터셋 구축 가능.

### Engineering Lead

**S (강점):**
- Pure FFmpeg 아키텍처 — Pydub 없이 메모리 누수 위험 제거
- Sequential Acrossfade 필터 그래프 — 수학적으로 정확한 크로스페이드
- Fail Fast Validation + Provenance — SHA-256 해싱, 재현성 보장
- 도메인 특화 자동화 — YouTube Shorts + Metadata end-to-end

**W (약점):**
- 단일 파일 모놀리스 (1,900줄) — 테스트 어려움, 협업 병합 충돌
- 플랫폼 종속성 (macOS 전용) — ffmpeg-full 경로 하드코딩
- 에러 핸들링 부재 — 부분 복구/재시도 로직 없음
- 확장성 제약 — 웹 UI, 병렬 처리, 클라우드 인코딩 통합 어려움

**O (기회):**
- 모듈화 리팩토링 — core/generators/cli 패키지 분리 + pytest
- 컨테이너화 (Docker) — 플랫폼 종속성 해결, 클라우드 배포
- CLI → API 전환 — FastAPI 래퍼, 웹 대시보드
- FFmpeg 필터 라이브러리 오픈소스화

**T (위협):**
- FFmpeg 버전 의존성 — drawtext/subtitles 필터 호환성
- 1인 개발 버스 팩터
- 기술 부채 누적 — 테스트 부재, 하드코딩
- YouTube API/정책 변화 — 메타데이터 스펙 변경

**핵심 인사이트:** 단기 Docker + pytest 도입, 장기 API 래퍼 확장. MASTER 문서 기반 AI 파이프라인 자동화로 10배 가속 잠재력.
